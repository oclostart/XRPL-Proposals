# Amendment Idea: XLS-103 Deterministic Source Tag

## 1. Abstract

This standard defines a simple, deterministic method for generating Source Tags on the XRP Ledger from domain names. The goal is to provide a consistent and reproducible way for applications, dApps, and services to derive Source Tags, improving traceability and interoperability across the XRPL ecosystem.

The method uses SHA-256 hashing followed by taking the first 4 bytes of the resulting hash, converted into a 32-bit unsigned integer.

## 2. Motivation

Currently, there is no standardized way for applications to generate Source Tags. Many projects choose arbitrary numbers, which makes it difficult to identify which application or service originated a transaction. A deterministic standard allows:

- Consistent Source Tags across different implementations
- Easier identification of transactions originating from specific applications
- Better analytics and on-chain attribution
- Reduced risk of accidental tag collisions between different projects

## 3. Specification

To generate a Source Tag from a domain, follow these steps:

1. Normalize the input string:
- Convert to lowercase
- Trim whitespace
- Remove common prefixes (https://, http://, www.)

2. Hash the normalized string using SHA-256.

3. Extract the first 4 bytes of the hash. (8 hex characters)

4. Convert those 4 bytes into a 32-bit unsigned integer.

const cryptoModule = require('crypto');

function domainToSourceTag(domain) {
    // Step 1: Normalize the domain
    let normalized = domain.toLowerCase().trim();
    
    if (normalized.startsWith('https://')) {
        normalized = normalized.slice(8);
    } else if (normalized.startsWith('http://')) {
        normalized = normalized.slice(7);
    }
    
    if (normalized.startsWith('www.')) {
        normalized = normalized.slice(4);
    }

    // Step 2: Create SHA-256 hash
    const hash = cryptoModule.createHash('sha256')
        .update(normalized, 'utf8')
        .digest();

    // Step 3: Take first 4 bytes and convert to unsigned 32-bit integer
    const sourceTag = hash.readUInt32BE(0);

    return sourceTag;
}

// Example usage
const tag = domainToSourceTag("myweb3.me");
console.log(tag); // 2357011303

## 4. Rationale

Using the first 4 bytes of a SHA-256 hash provides a good balance between:

- Simplicity
- Determinism
- Low probability of collisions for typical use cases
- Compatibility with the 32-bit SourceTag field on the XRPL

This approach is intentionally kept minimal to encourage wide adoption.

## 5. Backwards Compatibility

This standard does not modify the XRP Ledger protocol. It is an off-chain convention for generating Source Tags. Existing transactions and Source Tags remain fully compatible.

## 6. Test Case

- myweb3.me → 2357011303
- https://MyWeb3.me → 2357011303 (same after normalization)
- www.example.com → should produce a consistent value
- Different domains should produce different tags in the vast majority of cases

## 7. Security Considerations

This standard does not introduce any on-ledger changes or security risks. It is a method for generating consistent metadata, Source Tags.

Projects should be aware that while collisions are unlikely, they are theoretically possible. For higher assurance, projects may choose to include a namespace in the input string.
